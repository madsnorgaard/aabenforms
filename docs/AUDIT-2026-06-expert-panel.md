# AabenForms - expert panel audit (June 2026)

Five independent expert lenses (Danish public-sector domain & compliance, security/GDPR, ECA workflow correctness, Drupal architecture & DevOps, accessibility/WCAG & product) reviewed the codebase read-only and grounded findings in real file paths. This report is the synthesis.

**Overall maturity:** Early / pre-pilot POC in active hardening. Engineering architecture is solid (modular DI, strong test volume, real CI for unit+kernel), but production-blocking gaps span identity trust (unverified JWT signatures, open PII endpoint), Danish compliance subsystems (no GDPR retention/erasure, no protected-person handling, no live Serviceplatformen certs), and unenforced ECA/accessibility correctness. Not deployable to a real kommune until the critical identity-forgery and PII-disclosure flaws are closed and the compliance subsystems are built. Honest self-documentation (PROMISES-VS-VERIFIED.md) is itself a maturity signal and should be preserved.

## Executive summary

AabenForms is a well-architected, modular Drupal 11 + Nuxt 3 e-forms platform with a genuinely strong skeleton: clean dependency-injected service boundaries across six custom modules, 18-19 templated Danish municipal ECA flows, a consistent MitID/CPR fail-closed identity-gating pattern, CPR encryption-at-rest design, hashed audit logging, and ~17K lines of test code. Its standout asset is the honesty of backend/docs/PROMISES-VS-VERIFIED.md, which truthfully maps claims to WORKS/PARTIAL/BROKEN/MISSING/UNVERIFIED/FABRICATED status and lets sales copy stay defensible in front of a kommune evaluator. The June 2026 hardening sweep closed real holes (consent gate now fail-closed with real CPR fields, MitID validation fails closed, token resolution fixed so CPR/CVR lookups actually call Serviceplatformen).

However, all five lenses independently converge on the same verdict: this is a pre-pilot POC, not production-ready, and three classes of defect block any live deployment. First, security trust is forgeable end-to-end: MitIdCprExtractor (aabenforms_mitid/src/Service/MitIdCprExtractor.php:184-204) decodes the JWT payload with NO signature verification, the /mitid/session/{session_id} route ships _access: 'TRUE' (unauthenticated PII disclosure to anyone holding a workflow_id), and workflow_id lives in localStorage (XSS-exfiltratable). Second, Danish-compliance subsystems that a DPO will demand simply do not exist: no GDPR retention/erasure, no protected-person (beskyttet person) rejection, no offentlighedsloven case lifecycle with deadlines, and unverified field-level CPR encryption. Third, structural ECA correctness is unenforced: caseworker_review_flow.yml has illegal successors: on its conditions (verified at lines 32/42) AND uses the wrong eca_scalar contract (operator '==' / left_value instead of operator equal / left + negate), making it doubly dead, with no modeler save-time schema validation to catch it. Compounding all of this, functional/JS and accessibility test suites never run in CI, so the WCAG 2.1 AA claim is unmeasured and ECA flows ship without end-to-end execution proof.

## High-value Danish case opportunities

- Byggesag (building permit) - already templated; high-value pilot because it exercises DAWA address prefill, SF1520 CPR verify, multi-stage caseworker review, and a 10-year retention schedule - a complete showcase of the platform.
- Offentlighedsloven FOI request as a real case lifecycle (not just routing): submitted/acknowledged/responded/appealed states, statutory acknowledgment + response deadlines on the Danish holiday calendar, cron-driven overdue escalation, and an ESDH/SBSYS journaling handoff - addresses a legally mandated workflow no proprietaere alternativ bundles cheaply.
- Parkeringstilladelse (parking permit) with live payment (Nets Easy / MobilePay) and Digital Post permit issuance - a self-contained, fee-bearing transaction that proves end-to-end MitID -> pay -> SF1601 letter with idempotency.
- Foraeldregodkendelse / dual-parent approval (boern med to foraeldre) - the parent_dual_approval flow is a differentiated Danish family-law case: two independent MitID branches, constant-time CPR match, both-approval audit; strong demo of correct identity binding once JWT verification lands.
- Adresseaendring and telefon-erklaering routed to municipal staff action (not auto-updating CPR/DR) with documented SLA - common, high-volume kommune cases that show responsible non-authoritative-register behaviour.
- NemLog-in Erhverv / business workflows (CVR + procuration) such as company verification and permit applications by virksomheder - opens the business-facing segment via SF1530 and procuration-to-role mapping, distinct from citizen MitID.
- MED-valg (employee/parent election) nomination + voting - the existing election service with CPR-hash double-vote prevention and timed voting windows is an unusual, sticky internal-governance case municipalities run annually.

## Findings by lens

### Danish public-sector domain & compliance (MitID/NemLog-in, eIDAS, Digital Post, Serviceplatformen, CPR/CVR validation & masking, GDPR, offentlighedsloven, kommune case typology)

*Maturity: early.* AabenForms is a solid pre-pilot ECA platform with genuine strengths in MitID OIDC (PKCE S256), CPR/CVR field encryption, audit logging, and 18 templated municipal workflows. However, it carries critical Danish-domain gaps: no protected-person (beskyttet person) handling or CPR masking in logs, missing GDPR data-retention/erasure subsystem, Digital Post lacks idempotency and recipient verification, NemLog-in Erhverv unimplemented, no ESDH/SBSYS integration, and field-level CPR encryption remains unverified in practice. The product has passed recent security hardening (consent gates now fail-closed, MitID validation fixed, token resolution corrected), but PROMISES-VS-VERIFIED.md reveals pre-production maturity. Real Danish deployment requires immediate work on retention policies, protected-person flows, and live Serviceplatformen certificates.

- **[critical] Protected person (beskyttet person) handling completely absent** (CPR/CVR validation & data protection)
  - Evidence: aabenforms_core/src/Service/AuditLogger.php line 101 comments 'protected_person' in status enum, but no upstream check in CprLookupAction (aabenforms_workflows/src/Plugin/Action/CprLookupAction.php lines 129-200) or MitIdCprExtractor (aabenforms_mitid/src/Service/MitIdCprExtractor.php). No workflow gate, no lookup rejection, no citizen notification. SF1520 returns status for protected persons but is never checked.
  - Fix: Implement protected-person detection in CprLookupAction.execute() and MitIdCprExtractor.extractPersonData(); gate workflows to deny submission and audit with status='protected_person_rejected'; document reference to CPR protocol section on person access rules.
- **[critical] GDPR data retention and right-to-erasure subsystem not implemented** (GDPR compliance (Databeskyttelsesloven))
  - Evidence: README.md line 65 states 'A retention and right-to-erasure subsystem does not exist yet and is tracked in issue #91.' AuditLogger.purge() (lines 235-253) only handles audit-log retention, not webform submissions, CPR fields, or user session data. PROMISES-VS-VERIFIED.md line 16 marks field-level CPR encryption and deletion-after-case-closure as 'UNVERIFIED / MISSING'.
  - Fix: Build aabenforms_gdpr module: (1) webform_submission hook to detect case-closure/workflow-completion events; (2) field deletion targeting encrypted CPR + related PII; (3) audit-log retention policy config; (4) drush command for manual purge; (5) test coverage for end-to-end deletion.
- **[high] Digital Post recipient CPR taken from request body (no session verification); no idempotency guard** (Digital Post (SF1601) security & reliability)
  - Evidence: PROMISES-VS-VERIFIED.md line 13 marks Digital Post as 'PARTIAL: Sends, but recipient CPR was taken from the request body and there is no idempotency (duplicate letters).' DigitalPostSender.send() (aabenforms_digital_post/src/Service/DigitalPostSender.php lines 37-66) generates transaction_id but does not store or check idempotency key. No verification that recipient CPR matches authenticated session CPR.
  - Fix: Inject MitIdSessionManager into DigitalPostSender; verify recipient CPR against authenticated session before calling client.send(). Add idempotency table (transaction_id -> result) to prevent duplicate sends on retry. Add integration test DigitalPostIdempotencyTest.php.
- **[high] NemLog-in Erhverv (business MitID) planned but not implemented** (Digital identity (eIDAS LoA for business users))
  - Evidence: README.md line 51 states 'NemLog-in Erhverv på vej'. aabenforms_mitid/src/Service/MitIdCprExtractor.php lines 147-150 extract CVR and business_name from token claims but no integration with business workflows. README.md modules table lists 'aabenforms_nemlogin' as 'Planned'. MitID is citizen-only; business entity applications (CVR-based) cannot authenticate.
  - Fix: Implement aabenforms_nemlogin module: separate OIDC client for Nida3 or NemLogin-Erhverv endpoint; CVR claim extraction + validation; role/authorization context from procuration/permissions; delegate aabenforms_mitid to act as citizen-only client.
- **[high] CPR masking in logs inconsistent; unmasked values leak in audit context JSON** (CPR/CVR masking & audit log privacy)
  - Evidence: MitIdOidcClient.completeFlow() (line 341) masks CPR in log message but AuditLogger.log() (aabenforms_core/src/Service/AuditLogger.php line 163) hashes identifier correctly. However, DigitalPostSender.send() (line 60) includes 'sender_cvr' unmasked in audit context JSON. CprLookupAction.execute() logs result token value. No central masking policy.
  - Fix: Add aabenforms_core/src/Service/CprMasker with mask() and isSensitive() methods; audit all logger.info/error calls in ECA actions and service classes; refactor AuditLogger.log() to mask context arrays before JSON encode; add test AuditLoggerMaskingTest.php.
- **[high] No offentlighedsloven (FOI) case retention framework; foi_request_flow is routing, not case lifecycle** (Danish public-records law (offentlighedsloven) and FOI case management)
  - Evidence: eca.eca.foi_request_flow.yml (config/sync) shows simple route to caseworker email without case creation, deadline tracking, or ESDH handoff. FoiRouteAction (aabenforms_workflows/src/Plugin/Action/FoiRouteAction.php) records '7 working days' deadline in step but does not create case entity or link to SBSYS. README.md line 82 lists 'SF1470 journaling #85' and 'ESDH adapters #86' as planned but unimplemented.

  - Evidence: CprLookupAction.demoModeAllowed() (aabenforms_workflows/src/Plugin/Action/CprLookupAction.php lines 67-73) silently falls back to demo data when no Serviceplatformen certificate is provisioned. Mark SendSmsAction as delegating to Beskedfordeler, not mocking.

### Security & Privacy / GDPR Auditor (Adversarial)

*Maturity: early.* AabenForms has strong identity gating and audit infrastructure, but three critical flaws block production use: (1) JWT signatures are never cryptographically verified, allowing identity forgery; (2) the /mitid/session endpoint has no access control, exposing CPR/PII to anyone with a workflow_id; (3) workflow_id is persisted to localStorage, shifting trust from HTTPOnly cookies to the JavaScript environment (XSS risk). Once these are fixed, the parent approval CPR gate and encryption at-rest design are solid.

- **[critical] JWT Signature Validation Not Implemented (Identity Forgery)** (MitID OIDC Token Validation)
  - Evidence: /home/mno/ddev-projects/aabenforms/backend/web/modules/custom/aabenforms_mitid/src/Service/MitIdCprExtractor.php lines 195-210: parseJwt() only base64-decodes payload; no cryptographic signature verification. composer.json has no firebase/jwt or JWT library. MitIdOidcClient::completeFlow() calls validateIdToken() which only checks exp/iat/aud claims, not signature.
  - Fix: Integrate firebase/jwt or lcobucci/jwt library. Verify RS256 signature against MitID JWKS endpoint before trusting any id_token claim. Add integration test that rejects forged tokens.
- **[high] Unauthenticated Session Data Endpoint (PII Disclosure)** (Session Endpoint Access Control)
  - Evidence: /home/mno/ddev-projects/aabenforms/backend/web/modules/custom/aabenforms_mitid/aabenforms_mitid.routing.yml lines 41-49: /mitid/session/{session_id} has _access: TRUE with no permission check. MitIdController::getSession() returns full CPR, name, email, birthdate, address to anyone who knows workflow_id.
  - Fix: Add CSRF token validation: include nonce in initial session, require it back on GET. Or use one-time authorization codes instead of long-lived workflow_ids.
- **[high] Workflow_ID Persisted to localStorage (XSS/Credential Theft)** (Frontend Session Storage)
  - Evidence: /home/mno/ddev-projects/aabenforms/frontend/stores/user.ts lines 62-66: setUser() stores workflow_id to localStorage; page reload fetches /mitid/session/{id}. XSS compromise grants 15-minute identity impersonation.
  - Fix: Use HTTPOnly session cookies for workflow_id instead. If cross-origin SPA requires bearer token, use one-time authorization code exchanged for short-lived JWT.
- **[medium] Nonce Validation Skipped if Empty (Replay Attack Risk)** (OIDC Replay Protection)
  - Evidence: /home/mno/ddev-projects/aabenforms/backend/web/modules/custom/aabenforms_mitid/src/Service/MitIdOidcClient.php lines 318-322: nonce validation uses if (!empty($options['nonce'])) — silently skipped if nonce not provided. Old id_token can be replayed.
  - Fix: Make nonce mandatory. Throw exception in completeFlow() if nonce missing or doesn't match. Never allow optional nonce validation.
- **[medium] CPR Encryption Fallback to Plaintext (Configuration Risk)** (CPR Encryption at Rest)
  - Evidence: /home/mno/ddev-projects/aabenforms/backend/web/modules/custom/aabenforms_core/src/Service/CprAccess.php lines 79-89: protect() logs error and returns plaintext CPR if encryption fails. aabenforms_core.install lines 177-179 warns if AABENFORMS_CPR_KEY missing but allows operation.
  - Fix: Fail HARD if encryption key missing or encryption fails. Do not silently store CPR unencrypted. Queue unencryptable submissions for manual review and abort.
- **[medium] Audit Log SHA-256 Hashing Without Salt (Rainbow Table Risk)** (Audit Log De-Anonymization)
  - Evidence: /home/mno/ddev-projects/aabenforms/backend/web/modules/custom/aabenforms_core/src/Service/AuditLogger.php line 163: identifier_hash is hash('sha256', $identifier) with no salt. Danish CPR has 1 billion possibilities; attackers can pre-compute offline.
  - Fix: Use hash_hmac('sha256', $cpr, $site_secret) where $site_secret is environment-backed. Prevents rainbow-table attacks.
- **[medium] No Default Audit Log Retention Policy (GDPR Compliance Risk)** (Audit Log Lifecycle)
  - Evidence: /home/mno/ddev-projects/aabenforms/backend/web/modules/custom/aabenforms_core/src/Service/AuditLogger.php purge() requires explicit retention_days param. No install hook creates cron job. Audit logs retained indefinitely, violating GDPR Article 5.
  - Fix: Set default audit log retention policy (90 days for transactional, 3 years for compliance holds). Register cron job in install hook to call purge() automatically.
- **[medium] Parent Approval CPR Verification Missing Rate Limiting** (Parent Approval Brute Force)
  - Evidence: /home/mno/ddev-projects/aabenforms/backend/web/modules/custom/aabenforms_workflows/src/Controller/ParentApprovalController.php mitidComplete() allows unlimited MitID login attempts per token.
  - Fix: Log failed CPR verification attempts. Rate-limit by submission_id + IP: reject after 3 failures in 5 minutes. Return 429 and notify case worker on threshold.
- **[low] Frontend Does Not Clear Expired Session from Memory** (Frontend Session Expiry Handling)
  - Evidence: /home/mno/ddev-projects/aabenforms/frontend/stores/user.ts restoreSession() clears localStorage on expiry but Pinia state retains stale sessionId. Expired workflow_id may be used in API calls.
  - Fix: Clear sessionId from Pinia state immediately on expiry detection, not just localStorage. Add explicit check before any API call requiring sessionId.
- **[low] Demo Mode Bypass Flag Not Prominently Warned** (Demo Mode Configuration)
  - Evidence: /home/mno/ddev-projects/aabenforms/backend/web/modules/custom/aabenforms_workflows/src/Plugin/Action/MitIdValidateAction.php allow_mitid_demo_mode config. If TRUE in production, action returns verified=true without real session.
  - Fix: Add warning banner in admin dashboard if demo mode enabled. Consider requiring explicit env var AABENFORMS_DEMO_MODE_ON to make accidental enablement harder.

### ECA workflow & modeler correctness engineering

*Maturity: early.* AabenForms has 19 ECA flows with well-designed MitID/CPR identity gating that follows a consistent fail-closed pattern. However, a catastrophic structural defect in caseworker_review_flow renders it completely non-functional: conditions have illegal `successors` fields (violating ECA schema), creating 2 dead conditions and 2 unreachable actions. The modeler_api integration lacks validation to catch such schema violations at save time. Token contracts are generally sound (actions set `<result_token>_status` companions for scalar gating), but test coverage is incomplete and workflow instantiation from templates lacks end-to-end execution verification.

- **[critical] Dead Flow: caseworker_review_flow structurally malformed conditions** (ECA schema validation & graph reachability)
  - Evidence: /home/mno/ddev-projects/aabenforms/backend/config/sync/eca.eca.caseworker_review_flow.yml:25-45 - Conditions check_both_complete and check_caseworker_pending declare successors: fields (invalid ECA YAML).
